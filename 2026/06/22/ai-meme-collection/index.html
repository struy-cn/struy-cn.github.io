自从 Andrej Karpathy 发明了「Vibe Coding」这个词，程序员的世界就回不去了。

Vibe Coding 的核心精神：不写代码，只写感觉。告诉 AI 你要什么，它写，你 review，跑通了就上线。写代码这件事正在变成一种……

氛围。

什么叫 Vibe Coding

用 Karpathy 的话说：「我完全不碰代码，只是不断跟 AI 说自己的想法，看它的输出，试着运行，把错误贴回去。」

这个说法在 HN 上引爆了讨论。最近一篇热帖叫《After two years of vibecoding, I’m back to writing by hand》（865 分，634 条评论），作者两年 vibe coding 后的结论是：手写更爽。

但这不妨碍 vibe coding 成为一个时代热词。就像「云原生」一样——不管你喜不喜欢，它已经来了。

梗图时间

1. Vibe Coding 的四个阶段

从聊天到让模型自己上线，你到了哪一级？

第一阶段：只会用 ChatGPT 问问题。
第二阶段：用 Claude Code / Cursor 写代码。
第三阶段：搭了一套 Agent 工作流，模型自己规划、执行、验证。
第四阶段：跟模型说一句话，代码写完、部署上线、通知发群里。

跨越每个阶段的标志，是你对「自己还需要写代码吗」这个问题答案的变化。

2. 本地模型 vs Claude Code

花一下午调本地模型：环境装好了、依赖通了一半、报了个看不懂的错。

然后转头用 Claude Code，一句话搞定。

不是说本地模型不好——隐私、离线、自定义都有价值。但在「快速出活」这件事上，差距确实明显。

3. 选择困难症

现在工具太多了，每天都冒出新的。

左边自己调参——深度可控，但费时间。右边 AI 全托管——快，但总觉得少了点什么。

程序员的一生，就是不断在「掌控感」和「效率」之间反复横跳的一生。

4. AI 不会替代程序员

这句话已经听过一百遍了。但后面半句才是重点：

「AI 不会替代程序员，但会用 AI 的会替代不会用的。」

这不是焦虑营销——这是效率阈值的抬升。当年用 IDE 的替代了用记事本的，现在用 AI 的替代了不用的。工具在变，规律不变。

5. 项目明天上线

Git 有 47 个未提交的改动。
CI/CD 还在跑第 3 次重试。
昨天承诺的功能——还没开始写。

但心态不能崩。This is fine.

6. IntelliJ IDEA vs Codex/Claude Code

Distracted Boyfriend 模板最适合这个场景：

IDEA 是老搭档——5 年了，稳定、可靠、什么活都能干。但最近你开始偷偷跟 Codex 和 Claude Code 搞在一起……

命令行跑起来是真的快，一句话生成整个模块，不用点鼠标。但每次它写完，你还是得默默打开 IDEA 来 review。

不是 IDEA 不好了，是 AI 命令行工具太会撩了。

7. 十年前 vs 现在

十年前：手写汇编，背指令集，拿十六进制排 bug。
现在：全靠 AI 补全写代码，但也不敢删那两行「好像有用但不知道干嘛」的注释。

工具进化从来都是把复杂的东西封装掉。以前你不需要懂 CPU 指令集也能写 Python，现在你不需要懂所有 API 细节也能 vibe coding。

8. 代码 Review 的真相

最后是全场最真实的一张。

当你 review 代码看到一坨离谱的东西：
—「这代码是 AI 写的吧？」
—「……一直都是。」

水平差的程序员写烂代码，水平好的程序员写的……也是烂代码，只是烂得更有架构感。

AI 至少不会在代码里写 // TODO: fix this later 然后一放就是三年。

HN 争论：Vibe Coding 到底行不行？

上面那篇《After two years of vibecoding, I’m back to writing by hand》在 HN 上炸出了 634 条评论（865 分），我把最有意思的观点整理了一下：

🤖 工具之争：Copilot 到底行不行？

一个被反复提起的话题：GitHub Copilot 算 vibe coding 吗？

Copilot 2021 年就有了，但那会儿就是个行内补全。真正的 vibe coding 被普遍认为始于 Claude 3.5（2024 年 7 月）——有用户吐槽 Copilot 体验：「用 Copilot vibe coding 的感觉，就像早期汽车设计师试图用缰绳驾驭新车。」（kaydub）

不过也有人为 Copilot 正名：「Copilot 现在也有 Planning 模式、Agent 模式、MCP 和 Tool Calling，完全可以在一个订阅下切换 Gemini、ChatGPT、Grok 和 Claude 来 vibe coding 小项目。」（Mashimo）

💊 医疗固件工程师的真实体验

一个做医疗固件的程序员说了一段很真实的话：

「我让 AI 给我的医疗器械固件写单元测试。AI 30 分钟生成的初稿大概能省我一天的工作量，但你要花 3 天时间去修复和打磨。关键是——写测试的过程就是我发现代码 bug 的过程。如果 AI 写的测试全通过了……它很可能只是在骗你说代码没问题。」
— popoflojo

这个视角很值得玩味：vibe coding 的效率红利是真实的，但代价是丢失了「写代码即理解代码」的学习曲线。

🧠 高级研发 vs 初级开发

另一个高赞评论直戳要害：

「AI 目前只擅长基础任务，哪怕你全程手把手喂。它的唯一用途就是一个付不起初级开发的资深开发雇来的初级开发。」
— Grimblewald

这解释了为什么经验丰富的人对 vibe coding 的体验两极分化——你能快速补全知道该怎么写的脚手架，但让你从头搭一个陌生的复杂系统时，AI 就完全不够看了。

📝 实用技巧：用规则文档驯服 AI

当然也有真正用好 vibe coding 的人分享方法：

「我会写一份 tests.md 来定义测试规范。每次 Claude 写完代码，我就说『在审查之前，确保所有测试遵循 @tests.md』。放在系统 prompt 里离远了效果会差，但对话里主动调用的效果非常好。」
— krschacht

这大概是 vibe coding 最实用的建议：不要只依赖模型本身的泛化能力，显式地把自己的规范和偏好写成一个文档，反复喂给它。

💡 速度悖论

有意思的是，评论区还出现了一个「速度悖论」——有人质疑「你花这么多时间跟 AI 指示和调试，还不如自己写快」；但立刻有人反驳：「我让它写个播放列表生成器，5 轮迭代就调好了，自己写的话至少要翻好几倍的时间。」

两种体验可能都对，区别在于：你的领域模型里有没有这个问题的模板。 有模板 → AI 飞快；没模板 → 纯手写更快。

🌐 企业级的现实

最后是一个残酷的现实：在 SSO 认证的企业环境中，Copilot 被锁得死死的。想接 Claude Code？逆向工程认证接口？在这些公司是「当场被开除」级别的操作。个人开发者跑得飞快，大型企业还在原地踏步。

写在最后

Vibe Coding 这个词之所以火，是因为它戳中了一个真实的变化：写代码的方式确实在变。

以前写代码是「翻译」——把脑子里的逻辑翻译成语言。
现在写代码是「对话」——跟 AI 描述你想要的东西，它来翻译。

以前是 you are what you code。
现在是 you are what you prompt。

工具会变，梗图不会。项目快上线的时候还是会慌，review 到烂代码还是会血压高，看到新工具还是会手痒。

这可能就是程序员不变的浪漫。 🦐
